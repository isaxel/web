# Jakarta EE --- Краткий конспект (формализованная версия)

## 1. Платформа Jakarta EE

Jakarta EE представляет собой стандартизированную платформу для
разработки корпоративных приложений на языке Java. Платформа определяет
набор спецификаций (API), описывающих контракт взаимодействия между
приложением и сервером приложений. Ключевые цели платформы включают
переносимость, масштабируемость, безопасность и унификацию архитектурных
подходов.

Современные версии Jakarta EE ориентированы на использование Java SE 17+
и облачные сценарии развертывания. Архитектура платформы основана на
принципе разделения спецификаций и реализаций: спецификации описывают
поведение, а серверы приложений обеспечивают их выполнение.

### Профили платформы

-   **Full Platform** --- полный набор спецификаций для сложных
    корпоративных систем.
-   **Web Profile** --- подмножество API, ориентированное на
    веб‑приложения.
-   **Core Profile** --- минимальный набор спецификаций для
    микросервисных архитектур.

## 2. Inversion of Control (IoC)

IoC является архитектурным принципом, при котором управление созданием и
конфигурированием объектов передаётся контейнеру. Объекты не создают
зависимости напрямую, а получают их извне, что снижает связанность
компонентов и повышает тестируемость.

Основные формы реализации IoC:

-   Dependency Injection (внедрение зависимостей)
-   Dependency Lookup
-   Событийные механизмы взаимодействия

Применение IoC позволяет централизовать управление жизненным циклом
компонентов и инфраструктурными сервисами.

## 3. Contexts and Dependency Injection (CDI)

CDI --- спецификация Jakarta EE, реализующая IoC и управление
контекстами. CDI определяет механизм автоматического внедрения
зависимостей, области видимости компонентов и событийную модель.

### Основные возможности CDI

-   Внедрение зависимостей через `@Inject`
-   Контексты жизненного цикла (`@RequestScoped`, `@SessionScoped`,
    `@ApplicationScoped`)
-   Использование квалификаторов
-   Producer‑методы (`@Produces`)
-   Interceptor‑механизмы
-   События (`@Observes`)

Контейнер CDI отвечает за создание, проксирование и уничтожение
компонентов в соответствии с активным контекстом.

## 4. Location Transparency

Location Transparency означает, что взаимодействие с компонентом не
зависит от его физического размещения. Вызовы методов выполняются
одинаково как для локальных, так и для удалённых объектов. Контейнер
обеспечивает прозрачность за счёт прокси‑объектов и инфраструктуры
удалённого вызова.

Данный принцип упрощает масштабирование, распределённые вычисления и
миграцию компонентов между узлами.

## 5. Управление жизненным циклом компонентов

Контейнер Jakarta EE автоматически управляет жизненным циклом
компонентов. Основные этапы:

1.  Instantiation (создание экземпляра)
2.  Dependency Injection
3.  Инициализация (`@PostConstruct`)
4.  Использование компонента
5.  Уничтожение (`@PreDestroy`)

Поведение компонента определяется его типом и областью видимости.

## 6. Дескрипторы развёртывания

Дескрипторы развёртывания --- XML‑файлы конфигурации, позволяющие
управлять поведением приложения без изменения исходного кода.

Типичные дескрипторы:

-   `beans.xml` --- конфигурация CDI
-   `web.xml` --- параметры веб‑контейнера
-   `persistence.xml` --- настройки JPA
-   `ejb-jar.xml` --- конфигурация EJB

В современных приложениях приоритет отдается аннотациям, однако XML
остаётся механизмом переопределения и централизованной конфигурации.

## 7. Enterprise JavaBeans (EJB)

EJB --- компонентная модель для реализации бизнес‑логики с поддержкой
транзакций, безопасности и распределённого выполнения.

### Типы Session Beans

-   **Stateless Session Beans** --- не сохраняют состояние между
    вызовами.
-   **Stateful Session Beans** --- сохраняют состояние, привязанное к
    клиенту.
-   **Singleton Beans** --- единственный экземпляр на приложение.

Контейнер EJB обеспечивает транзакционность, пуллинг и конкурентный
доступ.

### EJB Lite и EJB Full

EJB Lite представляет собой упрощённое подмножество спецификации. EJB
Full включает расширенные возможности, включая удалённые интерфейсы и
stateful‑компоненты.

## 8. Работа с электронной почтой (Jakarta Mail)

Jakarta Mail API предназначен для отправки и получения электронных
сообщений. API оперирует следующими базовыми сущностями:

-   Session --- конфигурация почтового соединения
-   Message / MimeMessage --- представление письма
-   Transport --- механизм отправки
-   Store / Folder --- доступ к почтовым хранилищам
-   Multipart --- составные сообщения и вложения

API используется для реализации уведомлений, отчётности и интеграционных
сценариев.
